# Talon Protocol Specification

## Overview

The Talon Protocol defines the communication between the **panel** (central control) and **nodes** (workers that run apps). Communication is bidirectional over WebSocket. Every message is wrapped in a `Message` envelope, and every received `Message` must be acknowledged with an `ack` response.

**Connection flow:**
`node.register` → `node.sync` → `node.ready` → normal operation

---

## Models

### Message

The envelope for all protocol communication.

```json
{
  "type": "string",
  "correlation_id": "string",
  "payload": "<event-specific struct>"
}
```

| Field | Description |
|---|---|
| `type` | Event identifier (e.g. `"app.create"`) |
| `correlation_id` | UUID generated by the sender; responses echo the same ID |
| `payload` | Event body as defined per event type |

> Responses (`ack`, `app.state`, `node.ready`) must echo the `correlation_id` of the message that triggered them. For spontaneous node events (e.g. crash-triggered `app.state`), the node generates a new UUID.

---

### App

```json
{
  "id": "string",
  "deploy_id": "string | null",
  "name": "string",
  "repo": "string",
  "resources": {
    "memory": "integer (MB)",
    "cpu": "float (cores)"
  },
  "env": { "KEY": "VALUE" },
  "status": "stopped | starting | running | stopping | error | unknown"
}
```

**Status values:**

| Status | Description |
|---|---|
| `stopped` | Container stopped normally |
| `starting` | Container is initializing |
| `running` | Container is active and healthy |
| `stopping` | Container is shutting down |
| `error` | Failure; `reason` field will be populated |
| `unknown` | State could not be determined by the node |

## Events

All events are sent as a `Message`. The examples below show the full envelope. Where a payload inherits a model, fields are shown inline with a comment indicating the source.

---

### Handshake

#### `node.register` — Node → Panel

Sent by the node immediately on connection. Must include the authorization token in the HTTP `Authorization: Bearer <token>` header.

```json
{
  "type": "node.register",
  "correlation_id": "uuid-1",
  "payload": {
    "node_id": "string",
    "version": "string"
  }
}
```

The panel responds with an `ack`. If authorized, it immediately follows with `node.sync`.

---

#### `node.sync` — Panel → Node

Sent after successful authorization. Carries the current state of all apps assigned to this node.

```json
{
  "type": "node.sync",
  "correlation_id": "uuid-2",
  "payload": {
    "items": [
      {
        "app": {
          // Models.App
          "id": "string",
          "deploy_id": "string | null",
          "name": "string",
          "repo": "string",
          "resources": { "memory": "integer", "cpu": "float" },
          "env": { "KEY": "VALUE" },
          "status": "string"
        },
        "deploy": {
          // Models.Deploy — null if no deploy exists
          "id": "string",
          "app_id": "string",
          "buildpack": "string",
          "branch": "string",
          "commit": "string",
          "created_at": "ISO8601 timestamp"
        }
      }
    ]
  }
}
```

---

#### `node.ready` — Node → Panel

Sent after the node has processed `node.sync`, signaling it is ready for operation. Reports the observed status of each app.

```json
{
  "type": "node.ready",
  "correlation_id": "uuid-2",
  "payload": {
    "apps": [
      { "id": "string", "status": "string" }
    ]
  }
}
```

---

### App Lifecycle

#### `app.create` — Panel → Node

Registers a new app on the node. Does **not** start the app automatically.

```json
{
  "type": "app.create",
  "correlation_id": "uuid-3",
  "payload": {
    // Models.App
    "id": "string",
    "name": "string",
    "repo": "string",
    "resources": { "memory": "integer", "cpu": "float" },
    "env": { "KEY": "VALUE" },
    "status": "stopped"
  }
}
```

---

#### `app.update` — Panel → Node

Updates specific fields of an existing app. `keys` explicitly lists which fields are being changed to prevent accidental overwrites.

```json
{
  "type": "app.update",
  "correlation_id": "uuid-4",
  "payload": {
    "id": "string",
    "keys": ["env", "resources"],
    "data": {
      // partial Models.App — only fields listed in `keys`
      "env": { "KEY": "VALUE" },
      "resources": { "memory": "integer", "cpu": "float" }
    }
  }
}
```

---

#### `app.deploy` — Panel → Node

Triggers a new deploy. The node builds and starts the app, then emits `app.state` with the result.

```json
{
  "type": "app.deploy",
  "correlation_id": "uuid-5",
  "payload": {
    // Models.Deploy
    "id": "string",
    "app_id": "string",
    "buildpack": "string",
    "branch": "string",
    "commit": "string",
    "created_at": "ISO8601 timestamp"
  }
}
```

Expected response: `app.state` with `status: "running"` on success or `status: "error"` on failure.

---

#### `app.start` / `app.stop` / `app.destroy` — Panel → Node

Sends an action to an app. `app.destroy` removes the app and all its resources from the node and is irreversible.

```json
{
  "type": "app.start",
  "correlation_id": "uuid-6",
  "payload": {
    "id": "string"
  }
}
```

All three actions trigger an `app.state` response from the node.

---

#### `app.state` — Node → Panel

Emitted by the node whenever an app's state changes — whether caused by a panel command or an internal event (crash, OOM, etc.).

Uses the `correlation_id` of the triggering command, or a new UUID for spontaneous events.

```json
{
  "type": "app.state",
  "correlation_id": "uuid-6",
  "payload": {
    // Models.AppState
    "id": "string",
    "deploy_id": "string | null",
    "status": "stopped | starting | running | stopping | error | unknown",
    "reason": "string | null"
  }
}
```

---

#### `ack` — Bidirectional

Sent in response to every received message. Echoes the `correlation_id` of the message being acknowledged.

```json
{
  "type": "ack",
  "correlation_id": "uuid-1",
  "payload": {
    "status": "ok | error",
    "reason": "string | null"
  }
}
```

---

## Error Handling

- Invalid or malformed messages receive an `ack` with `status: "error"` and a `reason` describing the problem.
- Async failures (e.g. a deploy that fails mid-build) emit an `app.state` with `status: "error"` and `reason` populated.
- On reconnection, the node must restart the handshake from `node.register`.

---

## Event Reference

| `type` | Direction | Expected response |
|---|---|---|
| `node.register` | Node → Panel | `ack` + `node.sync` |
| `node.sync` | Panel → Node | `ack` + `node.ready` |
| `node.ready` | Node → Panel | `ack` |
| `app.create` | Panel → Node | `ack` |
| `app.update` | Panel → Node | `ack` |
| `app.deploy` | Panel → Node | `ack` + `app.state` |
| `app.start` | Panel → Node | `ack` + `app.state` |
| `app.stop` | Panel → Node | `ack` + `app.state` |
| `app.destroy` | Panel → Node | `ack` + `app.state` |
| `app.state` | Node → Panel | `ack` |
